# AgentRadar 面试问答

## 为什么用 LangGraph，而不是把所有步骤写进一个函数？

搜索包含需求解析、查询规划、外部工具、过滤、研究、评分和持久化。LangGraph 让每一步拥有明确输入输出，便于记录轨迹、隔离失败、限制循环和替换单个节点。项目同时保留顺序执行降级路径，确保缺少 LangGraph 依赖时仍可测试核心逻辑。

## 哪些步骤适合规则，哪些步骤适合模型？

去重、fork/归档过滤、时间条件、排除项、真实路径校验和指标计算有确定答案，优先用规则。语义需求理解和复杂仓库调查等级更适合模型增强，但模型输出必须经过结构化校验和真实证据约束。V1 已支持可选模型增强，模型不能创建新候选或绕过排除规则；离线基线仍只使用规则，保证 CI 稳定且成本为零。

## 如何避免推荐不存在的文件？

研究服务先读取 GitHub 目录树，再从实际 `blob` 路径中按入口、图、状态、节点、工具、API 和测试顺序选择阅读路径。生成层不能自由编写路径。固定评测还会把结果与原始树做集合比较，路径幻觉率必须为 0。

## 为什么热度分和质量分要分开？

短期 Star 增长只能说明关注度，不能证明工程质量。热度分使用 24 小时/7 天增长、增长率、活跃度和新鲜度；质量分使用 Agent 能力与工程完整度。两个分数并列展示，可以识别“突然热门但工程薄弱”和“质量高但暂时不热门”的项目。

## 继续筛选为什么不重新搜索？

首轮已经为 GitHub 搜索和深度研究付出主要成本。继续筛选读取同一会话的当前候选，并按追加条件重新初筛；同等级分析报告直接从数据库复用，只为缺失报告的目标补充研究。这样响应更快，也避免重复消耗 GitHub 配额。

## 外部 API 失败如何处理？

GitHub 客户端统一设置超时、有限重试、TTL 缓存和标准错误类型。多条搜索语句允许部分失败；只有全部搜索失败才终止会话。单仓库的 README、目录、依赖、Release 或 Issue 读取失败会降级为已有证据，不影响其他候选。模型客户端同样限制超时和重试；网络、限流、HTTP 或 Schema 错误只记录非致命错误码，并立即使用规则解析或初筛结果。

## 如何控制成本和无限循环？

搜索计划限制为 3 至 5 条查询，最多两轮搜索、60 个规则候选和 5 个研究目标；README、文件、目录树都有大小和数量限制。状态记录工具与模型调用数，工作流没有开放式无限反思循环。

## 数据为什么既有 Repository 又有 Snapshot？

Repository 表示仓库最新状态，适合详情与搜索；Snapshot 是不可覆盖的时间序列事实，用于计算 24 小时和 7 天增量。如果只保存最新 Star 数，就无法回答项目何时升温。

## 为什么 Docker 默认使用 PostgreSQL，本地仍支持 SQLite？

SQLite 让新开发者无需额外服务就能启动和跑测试；PostgreSQL 更适合并发、持久化和正式部署。SQLAlchemy 保持业务层一致，Alembic 管理正式数据库迁移，Compose 默认走 PostgreSQL。

## 项目目前有哪些边界？

V1 没有多用户权限、长期画像、实时 SSE、多副本任务 Worker 和联网模型评测。离线评测只能阻止确定性回归，真实 GitHub 搜索召回率、延迟、限流和模型费用仍需要发布前采样。系统也不会自动克隆或运行陌生仓库，这是有意保留的安全边界。
